Refactoring Plan v2: generator_grid_prune_face.py
Last updated: 2026-02-20

0. Mandate
- DRY first: one source of truth for parameters, policies, and I/O schema.
- No fallbacks in core logic.
- No compatibility shims kept only for migration convenience.
- Fail fast on invalid state or malformed input.

1. Hard Rules (must hold during every step)
- Rule A: Do not introduce fallback paths.
- Rule B: Do not swallow errors with broad handlers such as `except Exception`.
- Rule C: Keep run parameters defined in exactly one place (RunConfig).
- Rule D: Prefer structural moves over local patches if duplication remains.

  - target size: <= 180 lines.

5. Refactoring Phases

Phase 0: Baseline freeze and guardrails
- Capture baseline outputs for fixed presets.
- Add strict grep checks to block fallback reintroduction.
- Keep behavior metrics snapshots for max-depth 0 and 1 runs.

Phase 1: Remove fallback behavior (strict mode by default)
- Step 1.1: expand strictness (completed on 2026-02-20)
  - backward-compatible norm inference removed from `expand.py`.
  - normalized bounds are now required when expansion request exists.
  - inconsistent stats now raise `ValueError` (fail-fast).
- Step 1.2: prune_axes strictness (completed on 2026-02-20)
  - `edge_line_key` now returns Optional line key.
  - unknown-direction edges are skipped explicitly.
  - `prune_unknown_line_key_total` stat is recorded.
- Step 1.3: ray-hit strategy unification
  - completed on 2026-02-20.
  - `graph.py:first_hit_edge` now uses one deterministic strategy.
  - face/global mixed branching removed from the selection path.
- Step 1.4: run-request I/O strictness (completed on 2026-02-20)
  - broad `except Exception` removed from request loading.
  - invalid stored payload schema now raises `ValueError`.

Phase 2: DRY consolidation of orchestration
- Create `creasegen/app.py` and move current `run()` body there.
- Keep `RunConfig` as single source for:
  - staged search controls
  - result params payload
  - run request payload
- Remove duplicate kwargs assembly spread across modules.
- Status on 2026-02-20:
  - `run()` orchestration moved to `creasegen/app.py::run_app`.
  - `creasegen/runtime_context.py::RunAppContext` introduced for dependency wiring.
  - entrypoint `run()` now delegates to `run_app`.

Phase 3: Wrapper elimination
- Replace entrypoint-level wrapper forwarding with direct module ownership.
- Move remaining helper glue into owning modules.
- Rename `_xxx_impl` exports to final names where ownership is stable.
- Status on 2026-02-20:
  - entrypoint wrapper count reduced to 0 by moving logic to `runtime_pipeline.py`.
  - wrapper elimination completed for runtime ops module.
- Target achieved: wrapper count in runtime ops is 0.

Phase 4: CLI and I/O split
- Move parser construction to `creasegen/cli.py`.
- Move run-request persistence to `creasegen/run_request_store.py`.
- Keep main file as a minimal executable shell.
- Status on 2026-02-20:
  - parser definition moved to `creasegen/cli.py::build_parser`.
  - run-request load/append/save moved to `creasegen/run_request_store.py`.
  - entrypoint `main()` now uses these modules directly.

Phase 5: Search and stage decomposition
- Split `search.py` into:
  - candidate generation
  - action application
  - acceptance policy
- Split `stage_search.py` into:
  - expand planning
  - stage loop control
- Status on 2026-02-20:
  - `search.py` helper logic was split into `search_candidates.py`, `search_actions.py`, and `search_policy.py`.
  - `search.py` now orchestrates recursion while delegated modules own candidate/action/policy details.
  - `stage_search.py` auto-expand planning and apply logic were split into `stage_expand_planning.py` and `stage_expand_apply.py`.
  - `stage_search.py` now focuses on stage loop control.
  - `stage_search.py` introduced typed `StageSearchConfig` and `StageSearchDeps`, and `pipeline_search.py` now builds them once per run instead of threading long kwargs.
  - `stage_search.py` coarse-round/limit/auto-expand decision updates were moved to `stage_search_policy.py`.
  - `stage_search_kwargs` dict plumbing was removed between `app.py` and `pipeline_search.py`; typed `StageSearchDeps/StageSearchConfig` are now built by `stage_search_bindings.py` and passed directly.
  - duplicate `search_stage(...)` call wiring in `pipeline_search.py` was consolidated via a local helper.
  - tuple-heavy returns were replaced with typed results: `StageSearchResult` (`stage_search.py`) and `StagedSearchResult` (`pipeline_search.py`), and `app.py` now consumes named fields.
  - mutable stage work budget threading (`a_work/b_work/a_norm_work/b_norm_work`) was replaced with `StageWorkBudget` across `stage_search.py`, `pipeline_search.py`, and `app.py`.
  - `seed_expand_initial_graph` tuple return was replaced with typed `SeedExpandResult`, and seed budget threading now reuses `StageWorkBudget`.

Phase 6: Cleanup and contract lock
- Remove dead code and migration-only aliases.
- Freeze external contracts (`RunConfig`, result payload schema).
- Document invariants and strict behavior.
- Status on 2026-02-20:
  - `result_payload.py` now locks output schema via `RUN_RESULT_KEYS` + fail-fast validation.
  - `stage_search_bindings.py` now uses explicit field-name constants for stage dep/config binding contracts.
  - stage/seed/result modules now publish explicit contract surface via `__all__` in `stage_search.py`, `pipeline_search.py`, `stage_search_bindings.py`, and `result_payload.py`.

10. Invariants (locked)
- Stage budget contract:
  - `StageWorkBudget` is the single mutable-budget carrier for stage/seed search.
  - stage-level budget threading by raw tuple fields (`a_work/b_work/a_norm_work/b_norm_work`) is not allowed.
- Stage result contract:
  - `search_stage` returns `StageSearchResult`.
  - `run_staged_search` returns `StagedSearchResult`.
  - `seed_expand_initial_graph` returns `SeedExpandResult`.
- Binding contract:
  - `stage_search_bindings.py` uses explicit `STAGE_SEARCH_DEP_FIELDS` and `STAGE_SEARCH_CONFIG_FIELDS`.
  - missing bindings fail fast via `AttributeError`.
- Result payload contract:
  - `result_payload.py` output keys must exactly match `RUN_RESULT_KEYS`.
  - schema mismatch fails fast via `ValueError`.
- Strictness contract:
  - no fallback markers in active modules.
  - no broad `except Exception` in active modules.
