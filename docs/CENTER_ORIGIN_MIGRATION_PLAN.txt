Center-Origin Migration Plan (Dyadic Offset)

Purpose
- Migrate the coordinate system to a center-based square: local domain [-1/2, 1/2] x [-1/2, 1/2].
- Support configurable origin offset originOffset=(dx,dy) where dx,dy are dyadic numbers (a+b*sqrt(2))/2^k.
- Keep symmetry logic stable (y=x in local coordinates).

Backup Strategy
- Create full backup copies before modification.
- Backup target set:
  - webapp/ui/App.vue
  - webapp/ui/styles.css
  - webapp/src/engine/tiling.ts
  - webapp/src/engine/creasegen_graph.ts
  - webapp/src/engine/creasegen_graph_ops.ts
  - webapp/src/engine/creasegen_grid_utils.ts
  - webapp/src/engine/creasegen.ts
  - webapp/src/engine/types.ts
- Backup destination:
  - webapp/_backup/center_origin_migration_<timestamp>/

Coordinate Specification
- Local coordinates: [-1/2, 1/2].
- World coordinates: world = local + originOffset.
- Inverse transform: local = world - originOffset.
- originOffset uses exact dyadic representation (no float-only source of truth).
- Engine remains the single source of truth for addability checks and bounds checks.
- UI must not own a separate acceptance rule; it only forwards edits and renders state.

Engine Responsibility Rule
- All validity checks (in-paper, lattice-valid, expand-eligibility) are decided in engine code.
- If coordinate normalization is used internally, normalization must happen inside engine entry points.
- Boundary definitions must be derived from originOffset consistently for all stages.

Implementation Milestones
1) Utility layer
- Add dyadic offset type and exact point add/sub helpers.
- Add centered boundary checks (inCenteredSquare).

2) Tiling migration
- Replace [0,1] clamps and penalties with [-1/2,1/2].
- Update corner-hit targets to centered square corners.
- Update symmetry projection and perturbation ranges.
- Update lattice candidate generation to centered domain.

3) Creasegen migration
- Update grid enumeration bounds from [0,1] to centered domain.
- Update square boundary initialization to centered corners.
- Update default diagonal seed to centered diagonal endpoints.
- Update square-membership checks.

4) UI migration
- Unify canvas/input/display to centered coordinates.
- Add originOffset(dx,dy) input using dyadic (a,b,k).
- Keep snapping on dyadic lattice in centered domain.

5) I/O compatibility decision
- If cp_graph_v1 compatibility is required, add boundary adapters only at import/export.
- Internal engine should remain center-based consistently.

6) Validation
- Run typecheck:web, typecheck:engine, build.
- Run regression on symmetry cases (axis/side) and seed segments.
- Compare behavior before/after with representative datasets.

Risks / Notes
- Partial migration causes subtle asymmetry bugs.
- Center-origin migration must include tiling, not only UI.
- Prefer exact dyadic operations where possible to avoid drift.

Execution Order
- Backup -> Utility layer -> Tiling -> Creasegen -> UI -> Compatibility -> Validation -> Docs.

Owner Decision Log
- dyadic originOffset selected by user.
- Consistent center-based processing preferred over per-call coordinate conversion.
